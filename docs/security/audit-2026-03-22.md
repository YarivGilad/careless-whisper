## Static Code Review

### 1. Command Injection (Linux shell-out)

## [Low] Linux shell-out window ID not sanitized before xdotool

**File:** src-tauri/src/output/paste.rs:225

**What it is:** The app passes a window ID string directly to `xdotool windowactivate` without validating that it contains only digits. The window ID comes from `xdotool getactivewindow` (an integer), not from user input or transcribed text — so in normal operation there is no injection risk.

**Scenario:** Only exploitable if an attacker has already compromised the system and can manipulate `xdotool`'s output. No real-world attack path from audio or network input. The transcribed text never reaches any shell command — it only reaches:
- `copy_to_clipboard()` using the `arboard` library (no shell)
- Emitted as `transcription-complete` IPC event (Tauri, no shell)
- Pasted via Ctrl+V keystroke simulation (hard-coded, not user text)

**Fix:** Low priority. Optionally validate that `window_id` contains only ASCII digits before passing it to `xdotool`. Example:

Before:
```rust
let focus_status = std::process::Command::new("xdotool")
    .args(["windowactivate", "--sync", window_id])
    .status()?;
```

After:
```rust
if !window_id.chars().all(|c| c.is_ascii_digit()) {
    return Err(format!("Invalid window ID format: {}", window_id));
}
let focus_status = std::process::Command::new("xdotool")
    .args(["windowactivate", "--sync", window_id])
    .status()?;
```
### 2. IPC Input Validation

## [Medium] Path traversal in download_model and delete_model

**File:** src-tauri/src/models/downloader.rs:46-56 (URL + path construction), src-tauri/src/commands.rs:218-225

**What it is:** The `model` parameter (a string like "base") is inserted directly into a file path (`ggml-{name}.bin`) and a HuggingFace URL without validation. A value like `../../etc/cron.d/evil` would construct a path outside the models directory. Rust's `PathBuf` does not reject `..` components automatically; they are normalized only when calling `.canonicalize()`, which the code does not do.

**Scenario:** If the webview is ever compromised (e.g. via XSS — see CSP finding), an attacker could call `download_model("../../some/path")` to write a file outside the models directory, or `delete_model("../../config")` to delete the app's config file. The model parameter is also used in the HuggingFace URL, so a value like `"base?token=xxx"` could append query parameters (though this has less impact since the URL is controlled by the app). Without a CSP bypass this is not directly reachable from outside the machine.

**Fix:** Add an allowlist check before using the name. The valid model names are a fixed set of 5 strings defined in `MODELS` at lines 15–21. Reject anything not in that list:

Before:
```rust
#[tauri::command]
pub async fn download_model(app: AppHandle, model: String) -> Result<(), String> {
    downloader::download_model(app, model).await
}
```

After:
```rust
const VALID_MODELS: &[&str] = &["tiny", "base", "small", "medium", "large-v3"];

#[tauri::command]
pub async fn download_model(app: AppHandle, model: String) -> Result<(), String> {
    if !VALID_MODELS.contains(&model.as_str()) {
        return Err(format!("Unknown model: {}", model));
    }
    downloader::download_model(app, model).await
}
```

Apply the same check in `delete_model` (line 223). Also add it to `set_active_model` (line 228) — note that `set_active_model` already checks `model_path.exists()` before loading, which prevents loading non-existent paths, but the allowlist is still worth adding to be consistent and prevent path traversal to existing files outside the models directory.
### 3. CSP Disabled + get_recent_logs

## [Medium] Content Security Policy is disabled

**File:** src-tauri/tauri.conf.json:40

**What it is:** `"csp": null` disables the Content Security Policy for the webview. A CSP is a safety net that prevents injected scripts from running even if someone manages to sneak HTML into the page. Without it, any future code that renders user-controlled text as HTML (using `dangerouslySetInnerHTML` or direct DOM writes) would immediately allow script execution.

**Scenario:** Currently the React components use safe text rendering, so there is no active XSS path. But the missing CSP means one careless line of code in the future could create a serious vulnerability with no safety net. Combined with the path traversal vulnerability in `download_model` / `delete_model`, if those commands were ever called from a compromised frontend, the attacker would have stronger code execution guarantees.

**Fix:** Add a restrictive CSP to tauri.conf.json:

Before:
```json
"csp": null
```

After:
```json
"csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src https://huggingface.co"
```

Note: `'unsafe-inline'` is needed for Tailwind's inline styles. The `connect-src` allows model downloads from HuggingFace.

## [Info] get_recent_logs returns raw log content including transcriptions

**File:** src-tauri/src/commands.rs:355-362

**What it is:** The `get_recent_logs` command returns the last 100 lines of the app's log file as a plain string to the frontend. Log lines may include transcribed speech and system paths. Currently this text is only written to the clipboard (safe), but it is worth noting as a data exposure surface: anyone with access to the clipboard after a "Report Issue" action will see recent transcriptions.

**Scenario:** User presses the "Report Issue" button in the Settings window → logs are copied to clipboard → user pastes them into a GitHub issue → any private speech that was transcribed recently appears in the public issue. This is a privacy concern but not a security vulnerability, since the user is explicitly copying the logs.

**Fix:** Before the "copy to clipboard" action in the frontend, show the user the log content and ask them to review/redact it. No code change needed in Rust. In `src/components/Settings.tsx:reportIssue()`, add a modal or preview step:

Before:
```typescript
const reportIssue = async () => {
    const logs = await invoke<string>("get_recent_logs");
    await navigator.clipboard.writeText(logs);
    setLogsCopied(true);
    // ...
};
```

After:
```typescript
const reportIssue = async () => {
    const logs = await invoke<string>("get_recent_logs");
    // Show user a preview of logs and ask for confirmation
    if (window.confirm("Copy logs to clipboard? Recent transcriptions will be visible.\n\n" + logs.substring(0, 200) + "...")) {
        await navigator.clipboard.writeText(logs);
        setLogsCopied(true);
    }
    // ...
};
```

Or use a modal component for a better UX.

### 4. Unsafe Rust

## [Info] Unsafe blocks are well-bounded and follow platform API conventions

**Files:** src-tauri/src/output/paste.rs, src-tauri/src/lib.rs, src-tauri/src/commands.rs, src-tauri/src/audio/volume.rs, src-tauri/src/audio/capture.rs

**What it is:** The codebase contains ~16 unsafe blocks across 5 files, all calling platform C/system APIs:
- **macOS (Objective-C & CoreGraphics):** objc2 message sends to NSWorkspace/NSRunningApplication (paste.rs:20–32), CoreGraphics CGEventCreateKeyboardEvent / CGEventPostToPid (paste.rs:75–93), CoreAudio AudioObjectGetPropertyData/SetPropertyData (volume.rs:50, 75, 92, 116, 134), and ApplicationServices AXIsProcessTrusted (commands.rs:261, lib.rs:56–70)
- **Windows (Win32):** GetForegroundWindow (paste.rs:105), SetForegroundWindow + SendInput (paste.rs:123–164), and COM-based audio endpoint volume (volume.rs:165, 183)
- **Linux (libc):** Single mkfifo syscall (lib.rs:100)
- **Cross-platform:** Trait impl marker `unsafe impl Send for RecordingHandle` (capture.rs:13)

**Assessment:** Each unsafe block is short and well-bounded. All raw pointer returns from C APIs are null-checked before use (CGEventCreateKeyboardEvent at paste.rs:77, 87; msg_send results at paste.rs:23, 27, 66). CFRelease usage follows Apple's standard pattern: the returned `*const c_void` from CFDictionaryCreate is cast to `*mut c_void` for release (lib.rs:68, commands.rs:314) — this is the idiomatic way to call CFRelease in Rust bindings. Windows COM methods return proper Result types via windows-rs, so error handling is automatic. The unsafe impl Send for RecordingHandle has a comment explaining that cpal::Stream is not Send on macOS but the single-thread usage makes it safe.

**Scenario:** No identified memory safety risks in the current unsafe code. The patterns used (objc2 message sends, CoreFoundation CFRelease, windows-rs COM bindings, libc system calls) are standard and idiomatic for their respective APIs. Pointer arithmetic and buffer handling are delegated to the OS-provided APIs, not done manually in unsafe code.

**Fix:** No changes needed. For future reference: any new unsafe block should be accompanied by a comment explaining *why* it's needed and *what invariants* make it safe (as seen in capture.rs:11–12).

### 5. Overly Broad Permissions / macOSPrivateApi

## [Info] Permissions are appropriate and justified

**Files:** src-tauri/Info.plist, src-tauri/tauri.conf.json:42, src-tauri/src/lib.rs:27-71

**What it is:** The app requests two macOS permissions via Info.plist:
- **NSMicrophoneUsageDescription:** Declares microphone access for voice recording.
- **NSAppleEventsUsageDescription:** Declares Accessibility permission for paste simulation.

Both are justified for core functionality. The app also sets `"macOSPrivateApi": true` in tauri.conf.json to enable Tauri's access to private macOS APIs, which is necessary for the transparent recording overlay window (line 33 of tauri.conf.json: `"transparent": true`).

**Scenario:** No over-privileged permissions identified. The app does not request camera, contacts, location, or full disk access. All three macOS permission use cases are:
1. Microphone: Required to capture audio for transcription.
2. Accessibility (AXIsProcessTrustedWithOptions): Required to simulate Ctrl+V paste via CGEventPostToPid on macOS (called from output/paste.rs). Without Accessibility permission, paste simulation fails and text is only copied to clipboard.
3. macOSPrivateApi: Required for the transparent overlay window. This is a Tauri framework feature for transparent windows on macOS.

**Fix:** No changes needed. Note for future maintainers: `macOSPrivateApi: true` would cause immediate rejection if the app were submitted to the Mac App Store. Keep this in mind if distribution plans change (e.g., moving from direct download to App Store distribution).

### 7. Settings Persistence

## [Info] Settings persistence is safe; config file is world-readable

**File:** src-tauri/src/config/settings.rs:71–78

**What it is:** Settings are saved as JSON using serde. The `save()` method serializes the `Settings` struct with `serde_json::to_string_pretty` and writes it via `std::fs::write`. No injection risks were found. The config file is written with default OS permissions (0o644 on Unix), meaning other users on the same machine can read it. The config contains hotkey preferences, language settings, and model choices — no passwords or tokens.

**Scenario:** On a shared machine, another user could read your hotkey binding (e.g., "CmdOrCtrl+Shift+Space") and language preference (e.g., "en"). This is low-sensitivity information. The only potential privacy concern is if the `active_model` choice reveals workflow (e.g., large model suggests longer transcriptions), which is minimal.

**Fix:** Optionally set 0o600 permissions (read/write for owner only) on the config file so only the current user can read it. This is not required unless you share your machine with untrusted users. If implemented, add after the `std::fs::write` call:

```rust
#[cfg(unix)]
{
    use std::os::unix::fs::PermissionsExt;
    let perms = std::fs::Permissions::from_mode(0o600);
    std::fs::set_permissions(&path, perms).map_err(|e| e.to_string())?;
}
```

### 8. Model Download Integrity

## [Medium] No integrity check on downloaded model files

**File:** src-tauri/src/models/downloader.rs:46–91

**What it is:** After downloading a model `.bin` file from HuggingFace (line 48), the app renames it from `.part` to `.bin` (line 88) and loads it into whisper.cpp without verifying its SHA256 hash. The download uses HTTPS (which prevents interception in transit), but does not verify the file matches the expected checksum.

**Scenario:** If HuggingFace were compromised, or if the download were intercepted via an HTTPS MITM attack (e.g. a compromised corporate proxy), a malicious `.bin` file could be loaded into whisper.cpp (a C library). A crafted model file could potentially exploit a memory safety bug in whisper.cpp's model parser, leading to code execution or data corruption.

**Fix:** After downloading, verify the SHA256 of the file matches the known-good hash before renaming from `.part` to `.bin`. The expected hashes are published in the ggerganov/whisper.cpp repository. Hard-code them alongside the model definitions in `downloader.rs`:

Before:
```rust
const MODELS: &[(&str, u32, u32)] = &[
    ("tiny", 75, 390),
    ("base", 142, 500),
    // ...
];
```

After:
```rust
const MODELS: &[(&str, u32, u32, &str)] = &[
    ("tiny", 75, 390, "expected-sha256-hash-here"),
    ("base", 142, 500, "expected-sha256-hash-here"),
    // ...
];
```

Then after download (around line 86), compute SHA256 of the `.part` file and compare before rename:

```rust
use sha2::{Sha256, Digest};

// After file.flush() and drop(file):
let mut hasher = Sha256::new();
let mut file = tokio::fs::File::open(&part_path).await.map_err(|e| e.to_string())?;
let mut buf = [0; 8192];
loop {
    let n = file.read(&mut buf).await.map_err(|e| e.to_string())?;
    if n == 0 { break; }
    hasher.update(&buf[..n]);
}
let computed_hash = format!("{:x}", hasher.finalize());
if computed_hash != expected_hash {
    std::fs::remove_file(&part_path).map_err(|e| e.to_string())?;
    return Err(format!("Model integrity check failed: hash mismatch"));
}
std::fs::rename(&part_path, &final_path).map_err(|e| e.to_string())?;
```

Add `sha2 = "0.10"` to `src-tauri/Cargo.toml`.

### 9. Linux FIFO Listener

## [High] Linux FIFO listener allows any same-user process to trigger recording

**File:** src-tauri/src/lib.rs:79–146 (mkfifo call at line 100)

**What it is:** On Linux, the app creates a named pipe (FIFO) at `~/.local/share/careless-whisper/careless-whisper.sock` with mode 0o644 (rw-r--r--). Any process running as your user account can write to this file to silently start or stop audio recording — without requiring the hotkey. The FIFO is designed this way intentionally (to support custom keybindings on Wayland), but it also means any compromised app running as you could trigger recording.

**Scenario:** Malicious software installed under your user account (a compromised npm package, a malicious VS Code extension, a trojanized dependency, etc.) could write to the FIFO, start a recording, wait for you to speak sensitive information, stop the recording, and read the transcribed text from the clipboard. For example:
```bash
# Attacker's script runs as your user:
echo "x" > ~/.local/share/careless-whisper/careless-whisper.sock  # start recording
sleep 10  # capture audio
echo "x" > ~/.local/share/careless-whisper/careless-whisper.sock  # stop recording
# Transcribed text is now in clipboard; malware reads it
```

**Impact:** The buffer is 128 bytes and its content is discarded (line 130: `let _ = file.read(&mut buf)`), so there is no command injection risk. However, an attacker gains silent audio access without the user pressing the hotkey, and can exfiltrate transcribed speech via clipboard.

**Important note on fixes:** FIFO file permissions (`0o644` vs `0o600`) control access from *other Unix users* (different UIDs). They do **not** prevent other processes running as **your own user** from writing to the file. Since the stated attack scenario involves malware running as the same user, the real fix is Option 2 (the secret token). Option 1 is hygiene only.

**Fix (Option 1 — Hygiene only, does not close the stated attack):** Change the FIFO mode from `0o644` to `0o600` to prevent other Unix user accounts from reading/writing the pipe. This does not help against same-user malware (the scenario above), but removes unnecessary world-read access:

Before (line 100):
```rust
let ret = unsafe { libc::mkfifo(fifo_c.as_ptr(), 0o644) };
```

After:
```rust
let ret = unsafe { libc::mkfifo(fifo_c.as_ptr(), 0o600) };
```

**Fix (Option 2 — Recommended: closes the stated attack):** Add a random secret token written to a `0o600` file on startup. Require the FIFO message to include the token. Other processes running as your user cannot read a `0o600` file they didn't create, so they can't learn the token:

```rust
// On startup: generate token
let token = format!("{}", uuid::Uuid::new_v4());
std::fs::write(data_dir.join("fifo.token"), &token)
    .map_err(|e| e.to_string())?;

// In FIFO reader loop (around line 130):
let mut buf = [0u8; 64];
let _ = file.read(&mut buf);
let received = String::from_utf8_lossy(&buf).trim().to_string();
if received != token {
    log::warn!("FIFO token mismatch, ignoring");
    continue;
}
// Token matches, proceed with toggle
```

Keybinding scripts that invoke the FIFO would then use:
```bash
echo "$(<~/.local/share/careless-whisper/fifo.token)" > ~/.local/share/careless-whisper/careless-whisper.sock
```

**Summary:** For a personal machine, Option 2 is the meaningful fix (~30 minutes). Option 1 alone gives false confidence for the attack described above.

## Recommended Fix Order

Fix these in order — highest impact first:

1. **[High] FIFO secret token** (`lib.rs`, FIFO reader loop): Implement the secret token (Option 2) to prevent same-user malware from triggering recording. ~30 minutes. This is the fix that actually closes the stated attack vector. The mode change (Option 1) alone is not sufficient.

2. **[Medium] Model name allowlist** (`commands.rs:218, 223, 228`): Add a 3-line allowlist check that rejects any model name not in the known set of 5. Closes path traversal and URL injection. ~10 minutes.

3. **[Medium] Enable CSP** (`tauri.conf.json:40`): Replace `"csp": null` with a restrictive policy string. ~5 minutes. Provides a safety net against future XSS and strengthens the path traversal defense.

4. **[Medium] Model download integrity** (`downloader.rs`): Add SHA256 verification after download using the `sha2` crate. Look up expected hashes from the ggerganov/whisper.cpp README. ~30-60 minutes.

5. **[Low] Validate window_id in paste_x11** (`paste.rs:225`): Optional. Only needed if you're specifically concerned about local attacker scenarios on Linux. Low real-world risk.

Everything marked **[Info]** requires no code changes.
